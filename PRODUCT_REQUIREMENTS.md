## 1. Product Purpose

Innovo Claude is an **AI-assisted funding document generation platform** built for Innovo Consulting. It helps consultants produce *Vorhabensbeschreibungen* ŌĆö formal German-language project description documents required for public funding program applications (e.g., WTT, BMBF, Eurostars).

The core problem it solves: writing funding applications requires synthesising company-specific facts, funding program rules, and a persuasive formal writing style. This process is slow, repetitive, and error-prone when done manually. The platform automates the first draft and provides an AI-assisted editing interface.

**Who uses it:** Internal Innovo consultants only. Registration is restricted to `@innovo-consulting.de` and `@aiio.de` email domains, enforced at the schema validation layer (`backend/app/schemas.py:UserCreate`).

**What it produces:** Structured multi-section Word documents (`.docx`) and PDFs in formal German funding application prose.

### 2.5 File Storage

All uploaded files (audio, PDFs, DOCX) are stored in **Supabase Storage** using hash-based deduplication.

- **Hash:** SHA-256 of file bytes (`file_storage.py:compute_file_hash`)
- **Path format:** `{file_type}/{hash[:2]}/{hash}.{ext}` (e.g., `pdf/ab/abcd1234....pdf`)
- **Deduplication:** `get_or_create_file()` checks `files.content_hash` before uploading
- **Max upload size:** 50 MB (enforced; returns HTTP 413 if exceeded)
- **Download:** `download_from_supabase_storage()` returns raw bytes; served via API endpoints

The `files` table is the canonical record. All other tables reference it by `file_id` (UUID FK).

## 3. Core Workflow

### 3.0 User-Facing Workflow (v2)

The user interacts with three steps only. All internal processing is automatic.

1. **Create Project** ŌĆö User provides: Company (select or name), Funding Program (select), Topic (one sentence). System creates a `Project` row, resolves the template, creates an empty `Document`, and triggers background context assembly.
2. **Context Assembly** ŌĆö System automatically crawls the company website, transcribes audio, extracts the company profile, loads funding rules, retrieves relevant past applications from the knowledge base, and performs domain research if no company data exists. User sees assembly status in the project workspace context panel.
3. **Generate and Refine** ŌĆö User clicks Generate in the workspace. System calls `PromptBuilder` with the assembled `ProjectContext`. User edits sections via chat. User exports the document.

Users do not manually connect Companies, Funding Programs, Templates, or Documents. These are internal data structures managed by the system.

---

### 3.1 Internal Processing ŌĆö Company Context

1. Company website URL and/or audio file provided at project creation (or fetched via research agent if absent).
2. Backend dispatches a **FastAPI `BackgroundTask`** (`process_company_background` in `companies.py`, also called by `context_assembler.py`).
3. Background task sequence:
   - Crawl website ŌåÆ `website_raw_text` ŌåÆ clean ŌåÆ `website_clean_text` (cached in `website_text_cache` by URL hash)
   - Transcribe audio via Whisper API ŌåÆ `transcript_raw` ŌåÆ clean filler words ŌåÆ `transcript_clean` (cached in `audio_transcript_cache` by file hash)
   - Run LLM extraction ŌåÆ structured `company_profile` JSON stored on `Company` row
4. `Company.processing_status` progresses: `pending` ŌåÆ `processing` ŌåÆ `done` / `failed`

### 3.2 Internal Processing ŌĆö Funding Program Guidelines

1. Admin creates a funding program and uploads guideline PDFs/DOCX (category `"guidelines"`).
2. On upload, text is extracted and cached in `document_text_cache` by file hash.
3. When guidelines change (combined file hash differs from stored `source_file_hash`), LLM extracts structured rules into `funding_program_guidelines_summary.rules_json`.
4. Rules cache is invalidated only when the set of guideline files changes.

### 3.3 Internal Processing ŌĆö Style Reference (System-Wide)

1. Admin uploads historical *Vorhabensbeschreibung* PDFs to the *Alte Vorhabensbeschreibung* module.
2. On "Generate style profile": combined hash of all source files ŌåÆ if new, LLM call ŌåÆ writes one row to `alte_vorhabensbeschreibung_style_profile`.
3. Only one style profile row is active at a time (keyed by `combined_hash`).
4. This profile is injected into every document generation call across the system.

### 3.4 Internal Processing ŌĆö Document Creation

1. Triggered automatically by project creation (not manually by the user in v2).
2. Backend resolves the template via `template_resolver.py`, creates a `Document` row with empty sections scoped to the project.
3. Editor enters **`reviewHeadings` mode**: user can rename, add, delete, and reorder section headings.
4. User clicks "Confirm Headings" ŌåÆ POST `/documents/{id}/confirm-headings` ŌåÆ `headings_confirmed = 1`.
5. Editor enters **`confirmedHeadings` mode**: user clicks "Generate Content".

### 3.5 Content Generation

1. POST `/documents/{id}/generate-content` triggers `_generate_batch_content()`.
2. In v2, context is read from `ProjectContext` via `PromptBuilder` (`services/prompt_builder.py`). If no `ProjectContext` exists (pre-v2 document, `project_id = NULL`), context is assembled directly from `Company` and `FundingProgram` rows (backward compat path).
3. Sections are processed in batches of 3ŌĆō5. Each batch is one LLM call. `milestone_table` sections are skipped.
4. Prompt block order: **Rules ŌåÆ Company ŌåÆ Topic/Domain ŌåÆ Examples ŌåÆ Style ŌåÆ Task** (load-bearing; do not change without testing).
5. Response is strict JSON: `{ section_id: "generated text", ... }`. Validated before saving. Retries up to 2├Ś on structural failure.
6. Editor enters **`editingContent` mode**.

### 3.6 Chat-Based Editing

1. User types an instruction in the chat panel (e.g., "Abschnitt 2.1 k├╝rzer fassen").
2. POST `/documents/{id}/chat` ŌåÆ backend parses the message to identify the target section.
3. `_generate_section_content()` is called with: current section content + user instruction (XML-delimited) + company context.
4. Response is **a suggestion only** ŌĆö not saved automatically. Frontend displays it with Approve / Reject buttons.
5. User clicks **Approve** ŌåÆ POST `/documents/{id}/chat/confirm` ŌåÆ section `content` updated; section `title` is never modified from suggestions.
6. User clicks **Reject** ŌåÆ no database write.

### 3.7 Export

User downloads the document as DOCX (via `python-docx`) or PDF (via `reportlab`) using `GET /documents/{id}/export?format=docx|pdf`.

## 5. LLM Pipeline

### 5.1 Model

All LLM calls use `gpt-4o-mini`. The model name is hardcoded at each call site. There is no model configuration layer or fallback. Knowledge base embeddings use `text-embedding-3-small` (separate model, embedding endpoint only).

### 5.1.1 Context Source (v2)

In v2, generation reads context from a `ProjectContext` snapshot pre-assembled by `context_assembler.py`. Context is assembled once at project creation and reused for all subsequent generation calls via `PromptBuilder` (`services/prompt_builder.py`).

Two new context sources are added in v2:
- `domain_research_json` ŌĆö structured domain and company research from `research_agent.py`
- `retrieved_examples_json` ŌĆö relevant past application excerpts from `knowledge_base_retriever.py`

If no `ProjectContext` exists (pre-v2 document, `project_id = NULL`), generation falls back to assembling context directly from `Company` and `FundingProgram` rows. This backward compatibility path must be preserved.

### 5.2 Call Sites

There are **six** LLM text generation call sites and **one** embedding call site:

| # | Function | File | Purpose | Temp | Max Tokens |
|---|----------|------|---------|------|-----------|
| 1 | `extract_company_profile()` | `extraction.py:135` | Extract structured facts from website + transcript | 0.0 | unlimited |
| 2 | `generate_style_profile()` | `style_extraction.py:95` | Extract writing style patterns from historical docs | 0.0 | 2,000 |
| 3 | `extract_rules_from_text()` | `guidelines_processing.py:110` | Extract structured rules from guideline PDFs | 0.3 | 4,000 |
| 4 | `_generate_batch_content()` | `documents.py:1203` | Generate initial section content | 0.7 | unlimited |
| 5 | `_generate_section_content()` | `documents.py:2212` | Edit existing section via chat | 0.7 | 2,000 |
| 6 | `_answer_question_with_context()` | `documents.py:2444` | Answer user questions about the document | 0.7 | 1,000 |
| 7 | `embed_for_retrieval()` | `services/knowledge_base_retriever.py` | Generate query embedding for pgvector semantic search | n/a | n/a |

### 5.3 Prompt Architecture

**Extraction calls (1, 2, 3):** Deterministic (`temperature=0.0` or `0.3`). JSON output enforced via `response_format={"type": "json_object"}` where possible. Inputs are system-generated content, not user text.

**Generation call (4) ŌĆö `_generate_batch_content()`:**

In v2, the prompt is assembled by `PromptBuilder` (`services/prompt_builder.py`) reading from `ProjectContext`. The block order is extended to six blocks:

```
=== 1. F├¢RDERRICHTLINIEN ===           ŌåÉ funding_rules_json (primary constraint ŌĆö always first)
=== 2. FIRMENINFORMATIONEN ===          ŌåÉ company_profile + website_text_preview
=== 3. PROJEKTTHEMA UND DOM├äNE ===     ŌåÉ topic + domain_research_json  (NEW in v2)
=== 4. REFERENZBEISPIELE ===            ŌåÉ retrieved_examples_json, max 3  (NEW in v2)
=== 5. STIL-LEITFADEN ===               ŌåÉ style_profile_json
=== 6. GENERIERUNGSAUFGABE ===          ŌåÉ list of section headings to generate
```

**This order is load-bearing.** Rules are injected first so the model treats them as the primary constraint. The v1 four-block sequence is preserved within the v2 six-block sequence (blocks 3 and 4 are inserted between Firmeninformationen and Stil-Leitfaden). Changing the order changes generation behaviour. Prompt wording within each block is unchanged.

Output: strict JSON `{ "section_id": "generated text", ... }`. Validated after every attempt. Retries up to 2 additional times on structural failure (3 total API calls worst-case).

**Edit call (5) ŌĆö `_generate_section_content()`:**

The system message explicitly labels the model as an *editor*, not an *author*. Prompt structure:
```
{style_guide}
SIE SIND EIN REDAKTEUR, KEIN AUTOR.     ŌåÉ role constraint
Aktueller Inhalt: {current_content}      ŌåÉ existing text (must exist)
Benutzeranweisung: <user_instruction>    ŌåÉ user instruction, XML-delimited
{instruction_text}
</user_instruction>
Firmeninformationen (NUR ZUR ST├£TZUNG):  ŌåÉ company context (support only)
{company_context}
```

Output: plain German text (no JSON). Markdown artifacts are stripped via regex post-processing.

**Q&A call (6) ŌĆö `_answer_question_with_context()`:**

Prompt structure:
```
KONTEXT:
{document_content}        ŌåÉ all section text concatenated
{website_summary}         ŌåÉ company website excerpt
{conversation_history}    ŌåÉ last 3 chat messages

Firmenname: {company_name}
BENUTZERFRAGE: <user_instruction>   ŌåÉ user question, XML-delimited
{user_query_text}
</user_instruction>
```

### 5.4 Prompt Injection Protection

User-controlled inputs (`instruction`, `user_query`) are wrapped in XML delimiters before injection into all prompts:

```xml
<user_instruction>
{user_input}
</user_instruction>
```

None guards are applied before wrapping: `instruction_text = instruction or ""`.

### 5.5 Token Logging

Before every LLM call, prompt size is logged:

```python
approx_tokens = len(prompt) // 4
logger.info("LLM {step} prompt size (chars): %s", len(prompt))
logger.info("LLM {step} prompt tokens: %s", approx_tokens)
```

Step labels: `batch generation`, `section edit`, `Q&A`, `company profile extraction`, `guideline extraction`, `style extraction`.

### 5.6 Input Truncation

| Input | Limit | Applied in |
|-------|-------|-----------|
| Company extraction (website + transcript combined) | 50,000 chars each | `extraction.py:78-79` (smart truncate: 60% head + 40% tail) |
| Style extraction (all docs combined) | 100,000 chars | `style_extraction.py:57-59` (head truncation) |
| Guideline extraction | 50,000 chars | `guidelines_processing.py:83` (inline slice in f-string) |
| Website text in generation prompt | 30,000 chars | `documents.py:~990` |
| Transcript text in generation prompt | 30,000 chars | `documents.py:~999` |

### 5.7 Caching Strategy

LLM calls are avoided by hash-based caching at two levels:

1. **Text extraction caches** (`audio_transcript_cache`, `website_text_cache`, `document_text_cache`): keyed on file content hash or URL hash. Same file uploaded twice ŌåÆ one Whisper call.
2. **Derived output caches** (`funding_program_guidelines_summary`, `alte_vorhabensbeschreibung_style_profile`): keyed on combined hash of all source files. Re-upload of the same files ŌåÆ no new LLM call.

## 6. Engineering Constraints

These constraints are hard rules for all modifications to this codebase.

### 6.1 From `CLAUDE.md` (mandatory)

1. **Never perform large refactors without explicit user approval.** Prefer small, incremental changes.
2. **Do not modify database schema unless explicitly requested.** Do not run migrations automatically.
3. **Do not modify environment variables.**
4. **Do not modify deployment scripts** (Render, Docker) unless instructed.
5. **Always follow the change workflow:** explain ŌåÆ propose diff ŌåÆ wait for approval ŌåÆ apply.
6. **Preserve API contracts.** Do not change response shapes or break existing endpoints.
7. **Avoid injecting unnecessary context into prompts.** Do not increase temperature or token limits without justification.
8. **Never log sensitive company data.** No prompt content, website text, or section text in log lines.

### 6.2 Database Constraints

- **Primary key types are mixed:** `users.email` is a String PK (not an integer). All user ownership checks use `user_email` string comparisons.
- **`headings_confirmed` is Integer (0/1)**, not Boolean, for SQLite compatibility. PostgreSQL also accepts this. Do not change the column type.
- **`content_json` is an unvalidated JSON blob.** Any code reading sections must defensively check for required keys (`id`, `title`, `content`). Missing keys have caused silent failures.
- **`chat_history` column was added in a late migration.** The `_safe_get_document_by_id()` function in `documents.py` has a three-strategy fallback to handle databases where the column may not exist. Do not remove this until the migration is verified in all environments.
- **No cascade deletes** are configured. Deleting a company does not delete its documents.

### 6.3 LLM Constraints

- **Do not change prompt wording without testing generation quality.** Prompts are in German. Even minor rewording can shift output tone.
- **Do not remove the `<user_instruction>` XML delimiters** from `instruction` or `user_query` injections.
- **Do not raise `max_tokens` limits** without justification. Current limits were set to bound cost.
- **Do not lower `temperature` below 0.7 for generation calls** (calls 4 and 5). These need creative variation; deterministic generation produces repetitive documents.
- **`milestone_table` sections must never be sent to the LLM.** The batch generator explicitly skips them at `documents.py:1068`. Do not remove this guard.
- **Section titles must never be updated from LLM suggestions.** The `/chat/confirm` endpoint updates `content` only. This is intentional ŌĆö headings are confirmed by the user and treated as fixed structure.

### 6.4 Frontend Constraints

- **All API calls must go through `src/utils/api.ts`.** Do not hardcode API URLs or create parallel fetch logic.
- **Do not add new global state** (new Contexts, global stores) without explicit approval.
- **`ProtectedRoute` must wrap all authenticated pages.** Do not bypass it.
- **Handle `AUTH_EXPIRED` explicitly** in components that call the API ŌĆö it is not auto-redirected centrally.
- **File uploads must preserve the multipart/FormData contract.** Do not set `Content-Type` manually on upload requests ŌĆö the browser sets the boundary automatically.

### 6.5 Security Constraints

- **Never make unrestricted external HTTP requests.** Website scraping in `website_scraping.py` has a known SSRF risk (no private IP blocklist). Any new external request code must validate the target URL against RFC 1918 ranges before connecting.
- **Never log prompt content, section text, company data, or user queries.** Log only metadata (IDs, lengths, status codes).
- **JWT secret must come from environment.** `JWT_SECRET_KEY` raises `RuntimeError` at startup if absent. Do not provide a default value.
- **Password reset tokens must not appear in API responses** in production. The dev-only token leak in `auth.py` must be removed before any production deployment.

### 6.7 Known Technical Debt

The following issues exist in the codebase and should not be accidentally made worse:

1. **`documents.py` is 3,300+ lines.** It contains document CRUD, company context formatting, prompt construction, batch generation, chat orchestration, section parsing (with 8 regex fallback strategies), Q&A, milestone extraction, and export. Future work should extract these into separate modules.
2. **No rate limiting on any endpoint.** Auth endpoints (login, password reset) and LLM endpoints are unrestricted per user.
3. **2-second frontend polling** on company processing status creates N ├Ś 0.5 read/s per concurrent user. There is no backoff and no push mechanism.
4. **SSRF in `website_scraping.py`.** User-supplied URLs are fetched without validating against private IP ranges. `allow_redirects=True` compounds this.
5. **Password reset token returned in API response** (dev-only code path in `auth.py`). Must be removed before production use.
6. **`content_json` is an unvalidated blob.** No Pydantic schema enforces the section structure at the API boundary.
