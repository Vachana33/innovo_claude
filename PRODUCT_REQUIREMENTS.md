## 1. Product Purpose

Innovo Claude is a **RAG-powered funding document generation platform** built for Innovo Consulting. It helps consultants produce *Vorhabensbeschreibungen* — formal German-language project description documents required for public funding program applications (e.g., WTT, BMBF, Eurostars).

The core problem it solves: writing funding applications requires synthesising company-specific facts, funding program rules, and a persuasive formal writing style. This process is slow, repetitive, and error-prone when done manually. The platform automates first drafts using retrieval-augmented generation and provides an AI-assisted chat editing interface.

**Who uses it:** Internal Innovo consultants only. Registration is restricted to `@innovo-consulting.de` and `@aiio.de` email domains, enforced at the schema validation layer.

**What it produces:** Structured multi-section Word documents (`.docx`) and PDFs in formal German funding application prose.

**Two roles:**
- **Admin** — populates the knowledge base (style references, company files, funding program guidelines)
- **User** — creates projects, confirms the template, generates documents, refines via chat

### 2.4 Database

**Development:** SQLite (`backend/innovo.db`, auto-created)
**Production:** PostgreSQL (Supabase), pool_size=5, max_overflow=10, sslmode=require
**Schema management:** Alembic, migrations in `backend/alembic/versions/`

### 2.5 File Storage

All uploaded files are stored in Supabase Storage using hash-based deduplication.
- **Hash:** SHA-256 of file bytes (`shared/file_storage.py`)
- **Deduplication:** `get_or_create_file()` checks `files.content_hash` before uploading
- The `files` table is the canonical file registry. All other tables reference it by `file_id`.

## 3. Core Workflow

### 3.0 User-Facing Workflow

Users interact with three steps. All internal RAG processing is automatic.

1. **Create Project** — Company name (or select), Funding Program (select), Topic (one sentence). Optionally: website URL, documents, audio.
2. **Context Assembly** — Backend automatically: scrapes the company website, transcribes audio, extracts company profile, loads funding rules, retrieves relevant past examples from the KB. User watches a three-step progress screen.
3. **Template Confirmation & Generation** — Progress screen completes, template preview opens. User edits template structure, clicks "Confirm template". Per-section RAG generation begins. User refines via chat.

---

### 3.1 RAG Ingestion — Admin Side

Every document or data source goes through: **EXTRACT → CHUNK → EMBED → STORE**

**Embedding model:** `text-embedding-3-small` (1536 dimensions). Never change this without a full re-embedding migration on all `knowledge_base_chunks` rows.

#### Source 1 — Alte Vorhabensbeschreibung
- Admin uploads PDF/DOCX files to `/alte-vorhabensbeschreibung/upload`
- Category tag: `alte_vorhabensbeschreibung`
- After EXTRACT → CHUNK → EMBED → STORE, an LLM call extracts a style profile (tone, structure, vocabulary, sentence patterns)
- Stored in `alte_vorhabensbeschreibung_style_profile` (keyed by combined hash of all docs)
- Injected into every generation system prompt to enforce Innovo's house style

#### Source 2 — Company Data
- Admin uploads company detail files to `/companies/{id}/documents/upload` (category: `company_detail`)
- User provides company website URL at project creation → scraped (category: `company_website`)
- Both go through EXTRACT → CHUNK → EMBED → STORE with `company_id` as `source_id`
- LLM call extracts structured company profile → stored in `companies.company_profile`

#### Source 3 — Funding Program Guidelines

Admin can add guidelines via **two paths**:

**Path A — URL scraping:**
- Admin provides a URL. Scraper handles: static HTML, JS-rendered/accordion content (expands hidden sections), linked PDF/DOCX files (downloads and extracts), sub-pages linked from the main URL.
- URL stored in `funding_program_sources` for monthly re-scrape
- Pipeline: EXTRACT → CHUNK → EMBED → STORE (source_type: `funding_guideline`)

**Path B — Direct file upload:**
- Admin uploads PDF/DOCX directly
- Standard file ingestion pipeline

Both paths trigger an LLM call to extract structured rules into `funding_program_guidelines_summary.rules_json` (eligibility, mandatory sections, word limits, deadlines).

**Only funding programs whose guidelines have been fully ingested appear in the user-facing dropdown.** Never expose programs that have not been processed.

#### Source 4 — User-Provided Context (at Project Creation)
- Document uploads → EXTRACT → CHUNK → EMBED → STORE (source_type: `user_upload`, source_id: `project_id`)
- Audio recording → Whisper transcription → CHUNK → EMBED → STORE (source_type: `audio_transcript`)
- Scoped to the project. Supplements — never replaces — the admin knowledge base.

---

### 3.2 Company Context Processing

1. Website URL and/or audio provided at project creation (or during context assembly via web search)
2. Backend dispatches a `BackgroundTask` (`process_company_background` in `services/companies/`)
3. Sequence:
   - Check `website_text_cache` by URL hash → if cache miss: crawl website → cache result
   - Check `audio_transcript_cache` by file hash → if cache miss: transcribe via Whisper → cache result
   - LLM extraction → structured `company_profile` JSON stored on `Company` row
4. `Company.processing_status` progresses: `pending` → `processing` → `done` / `failed`

---

### 3.3 Funding Program Guidelines Processing

1. Admin adds guidelines via URL or file upload
2. Text is extracted and cached in `document_text_cache` by file hash
3. Combined hash of all guideline files is compared to stored `source_file_hash`
4. If new or changed: LLM extracts rules → stored in `funding_program_guidelines_summary.rules_json`
5. Cache is invalidated only when the set of guideline files/content changes

---

### 3.4 Style Profile Processing (System-Wide)

1. Admin uploads historical Vorhabensbeschreibungen to `/alte-vorhabensbeschreibung/upload`
2. On "Regenerate style profile": combined hash of all source files → if new, LLM call → one row in `alte_vorhabensbeschreibung_style_profile`
3. Only one style profile row is active at a time (keyed by `combined_hash`)
4. This profile is injected into every generation call across the system

---

### 3.5 Context Assembly (Background Task)

Triggered automatically by project creation. Runs in five stages (see `SYSTEM_ARCHITECTURE.md §5`):

1. Company research (DB lookup → website scrape → web search fallback)
2. Funding guidelines loading
3. Template resolution
4. Knowledge base retrieval (vector search, top-10 chunks)
5. Style profile loading

Frontend shows progress screen reading from `assembly_progress_json`:
```
Step 1 — Extracting company data       [spinner → ✓]
Step 2 — Loading funding guidelines    [spinner → ✓]
Step 3 — Preparing document template   [spinner → ✓]
```

These are **informational steps**. No user confirmation is required at Step 1 or Step 2. When all three show ✓, the template preview opens automatically.

---

### 3.6 Template Confirmation

After the progress screen completes, the template preview opens showing the resolved section structure. User can:
- Add a section
- Delete a section
- Add a sub-heading
- Rename a heading
- Move a section up or down

User clicks **"Confirm template"** — this is the single user action that triggers generation.

---

### 3.7 Per-Section RAG Generation

After template confirmation, each section is generated via its own retrieval cycle:

1. **RETRIEVE** — vector similarity search scoped to `company_id + funding_program_id + project_id`. Query: section heading + description. Top-10 chunks.
2. **BUILD PROMPT** — system: style profile + funding guidelines constraints for this section. Context: retrieved chunks + company profile.
3. **GENERATE** — `gpt-4o-mini`, temperature 0.7, streamed response.
4. **STORE** — section content overwritten in DB (idempotent — never appends).

`milestone_table` section types are skipped — never sent to the LLM.

Each section generation is independently re-triggerable without side effects.

---

### 3.8 Chat-Based Refinement (RAG)

After generation, users refine via the right-column chat panel. Each message goes through a 5-step pipeline:

1. **PARSE** — identify referenced section, identify attachments
2. **INGEST** (if attachments) — EXTRACT → CHUNK → EMBED → STORE immediately, scoped to `project_id`
3. **RETRIEVE** — vector search scoped to `project_id + section_id`
4. **BUILD PROMPT** — style profile + section constraints + retrieved chunks + chat history
5. **GENERATE + UPDATE** — streamed response. If user accepts → section content overwritten in DB. If rejected → no write.

Full chat history is persisted in `project_chat_messages`. Older messages are summarised if the history grows too long.

---

### 3.9 Export

User downloads the document as DOCX (via python-docx) or PDF (via reportlab) using the export endpoint in `services/documents/router.py`.

## 5. LLM Pipeline

### 5.1 Models

| Purpose | Model |
|---------|-------|
| All text generation and extraction | `gpt-4o-mini` |
| Audio transcription | `whisper-1` |
| Chunk and query embedding | `text-embedding-3-small` (1536 dimensions) |

The embedding model name and dimension are constants. Never change without a full re-embedding migration.

### 5.2 LLM Call Sites (all in `backend/innovo_backend/`)

| # | Function | File | Purpose | Temp |
|---|----------|------|---------|------|
| 1 | `extract_company_profile()` | `shared/extraction.py` | Structured company facts | 0.0 |
| 2 | `generate_style_profile()` | `shared/style_extraction.py` | Writing style from historical docs | 0.0 |
| 3 | `extract_rules_from_text()` | `shared/guidelines_processing.py` | Rules from guideline text | 0.3 |
| 4 | `generate_section_content()` | `services/documents/service.py` | Per-section RAG generation | 0.7 |
| 5 | `refine_section_via_chat()` | `services/projects/chat_service.py` | Chat-based refinement | 0.7 |
| 6 | `embed_chunks()` | `services/knowledge_base/retriever.py` | Chunk embedding (ingestion) | n/a |
| 7 | `embed_query()` | `services/knowledge_base/retriever.py` | Query embedding (retrieval) | n/a |

### 5.3 Prompt Architecture

**Prompt block order (load-bearing — do not change without testing):**
```
1. FÖRDERRICHTLINIEN    ← funding guidelines (primary constraint — always first)
2. FIRMENINFORMATIONEN  ← company profile + website preview
3. REFERENZBEISPIELE    ← retrieved chunks for this section
4. STIL-LEITFADEN       ← writing style
5. GENERIERUNGSAUFGABE  ← section heading + user notes
```

**Prompt injection protection:** All user inputs wrapped in XML delimiters:
```xml
<user_instruction>
{user_input}
</user_instruction>
```

### 5.4 Token Logging

Before every LLM call, log prompt size (chars and approximate tokens). Never log prompt content, section text, company data, or user queries.

### 5.5 Input Truncation

| Input | Limit | Applied in |
|-------|-------|-----------|
| Company extraction (website + transcript) | 50,000 chars each | `shared/extraction.py` |
| Style extraction (all docs combined) | 100,000 chars | `shared/style_extraction.py` |
| Guideline extraction | 50,000 chars | `shared/guidelines_processing.py` |
| Website text preview | 500 chars | `project_contexts.website_text_preview` |

### 5.6 Caching Strategy

1. **Text extraction caches** — keyed on file hash or URL hash. Same file uploaded twice → one Whisper/extraction call.
2. **Derived output caches** — keyed on combined hash of all source files. Re-upload of same files → no new LLM call.
3. **URL cache TTL** — website text caches are considered stale after 30 days.

## 6. Engineering Constraints

### 6.1 Canonical Backend

- All active code belongs in `backend/innovo_backend/`
- `/backend/app/` is read-only legacy — never extend it
- Services import only from `shared/` — never from each other

### 6.2 Single Embedding Model

`text-embedding-3-small` is the only embedding model used across the entire system. Document it as a constant. Never change it without a full re-embedding migration.

### 6.3 Vector Search Scoping

Every vector search must filter by at least one scope: `company_id`, `funding_program_id`, or `project_id`. Global unfiltered search is forbidden.

### 6.4 Ingestion Idempotency

Before inserting new chunks for a source, delete all existing chunks for that `source_id`. Never append — always replace.

### 6.5 Generation Idempotency

Re-generating a section overwrites the existing content record. Never append to existing section content.

### 6.6 Cache-First Processing

Check website, audio, and document text caches before processing. Never re-process the same content if a valid cache entry exists.

### 6.7 Database Constraints

- `headings_confirmed` is Integer (0/1), not Boolean — SQLite compatibility. Do not change column type.
- `content_json` is an unvalidated JSONB blob. Code reading sections must defensively check for `id`, `title`, `content` keys.
- Pre-v2 documents have `project_id = NULL` — the v1 fallback path in generation must be preserved.

### 6.8 LLM Constraints

- Do not change prompt wording without testing generation quality. Prompts are in German.
- Do not remove `<user_instruction>` XML delimiters from user input injections.
- Do not raise `max_tokens` limits without justification.
- Do not lower generation temperature below 0.7 for calls 4 and 5.
- `milestone_table` sections must never be sent to the LLM.
- Section titles must never be updated from LLM suggestions — content only.

### 6.9 Security Constraints

- Never make unvalidated external HTTP requests. URL scraping must validate against RFC 1918 ranges.
- Never log prompt content, section text, company data, or user queries.
- JWT secret must come from environment — never provide a default.
- Password reset tokens must not appear in API responses in production.

### 6.10 Frontend Constraints

- All API calls must go through `src/utils/api.ts`.
- Do not add new global React state without explicit approval.
- `ProtectedRoute` must wrap all authenticated pages.
- Handle `AUTH_EXPIRED` explicitly in components that call the API.
